# Sync Plan: Backfill specs from the built project

The code in `dist/` is complete, deployed, and working in production. The specs in `src/` were left mostly unfinished during the build. Only the application spec was started, and it truncates mid-sentence partway through the data model section. Every brand and interface file is still scaffold placeholder text.

This sync proposes backfilling the specs to match what was actually built. No code will change.

## What will be updated

### Application spec (`src/app.md`)

Currently 49 lines and cuts off mid-sentence while describing the repositories table. I will rewrite this as the full spec covering the system that was built:

- The two-phase architecture: ingestion (fetch, filter, chunk, embed) and bible generation (Mapper agent for architecture overview, Deep-Dive agent per file).
- The four tables and what each one stores — repositories, files, code chunks (with embeddings as JSON arrays), and generated docs (with a discriminator for repo overviews vs. per-file deep-dives).
- The two independent lifecycle statuses on the repository row, ingestion and documentation, and what each value means.
- The five backend methods and the contract each one keeps with the frontend, including which ones run as fire-and-forget background tasks and which are synchronous.
- The hybrid AI routing decision: external OpenRouter for embeddings (since the SDK has no embedding action), native MindStudio Claude 4.6 Sonnet for text generation.
- The boilerplate filter rules, the chunking strategy, the concurrency controls, and the GitHub Git Data API flow used by the PR feature.
- The current out-of-scope list (concurrency in ingestion, concurrent-run guard on generation, fork-and-PR, retrieval-augmented deep-dive, auth, scenarios).

### Web interface spec (`src/interfaces/web.md`)

Currently lorem ipsum. I will replace it with a full description of the frontend that was built:
