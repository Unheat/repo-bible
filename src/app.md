An AI-powered documentation engine for GitHub repositories. The user points the app at a GitHub URL. The app fetches the repo's file tree, breaks each source file into semantically meaningful chunks, embeds those chunks for retrieval, and runs two AI agents over the result: a [Mapper]{High-level agent. Surveys the entire repo, identifies entry points, module boundaries, and cross-file relationships. Output is structural — what talks to what.} that produces a structural overview, and a [Deep-Dive]{Per-file or per-module agent. Reads chunks plus retrieved context to explain how a given piece works in detail.} that produces detailed per-file documentation. The end product is an interactive Codebase Bible that can be browsed in the web UI and published as a Pull Request on the source repository in one click.

~~~
This file is the locked spec covering Steps 1–5:

- Step 1: data model.
- Step 2: GitHub ingestion (URL parsing, tree fetch, filtering, file persistence).
- Step 3: async architecture, smart chunking, OpenAI-compatible embeddings, persistence.
- Step 4: Mapper + Deep-Dive agents.
- Step 5: GitHub PR integration + frontend UI (Ingestion view + Dashboard view).

Platform reality check, surfaced once at the top so it is not re-litigated downstream:

- The platform's database is managed SQLite, not PostgreSQL. The schema below is modeled relationally — every parent-child link is expressed as an ID column on the child — but enforcement of foreign keys and cascading deletes happens in application code, not via SQL constraints.
- Vector similarity search is not a native database operation. Embeddings are stored as a JSON array of numbers on the chunk row. Initial implementation computes cosine similarity in-process at query time. If/when scale forces a change, embeddings can be offloaded to an external vector store with the SQLite row holding only the external vector ID.
- The MindStudio SDK does NOT have an embedding action. Embeddings are generated by calling an OpenAI-compatible endpoint directly via the `openai` npm package (configured for OpenRouter via `OPENAI_BASE_URL`). Text generation IS handled by the SDK — Step 4's agents call `mindstudio.generateText`. This is the deliberate Hybrid Routing decision: external embeddings via OpenRouter, native MindStudio text generation, native MindStudio cost meter for the LLM passes.
- All four tables get auto-managed system columns (`id` UUID, `created_at`, `updated_at`, `last_updated_by`). They are not redeclared in the interfaces below.
~~~

## Data Model

The data model is strictly relational with four tables: `repositories`, `files`, `code_chunks`, and `generated_docs`. Each child row references its parent by ID, and deleting a parent must cascade to all of its descendants.

~~~
Cascading delete chain (enforced in application code via `removeAll` predicates inside `db.batch(...)`):

delete repository
→ delete all files where file.repoId = repository.id
→ delete all code_chunks where chunk.fileId IN (those file ids)
→ delete all generated_docs where doc.repoId = repository.id
(covers both repo_overview rows and file_deepdive rows in one shot,
since `repoId` is denormalized onto every generated_docs row)

A shared cascade-delete helper (e.g. `dist/methods/src/common/cascadeDeleteRepository.ts`) is the right home when re-ingestion gains a stale-row prune pass. Today no method removes a repository, so the helper is not yet implemented.

All cross-table reads in methods batch via `db.batch(...)` — there are no JOINs.
~~~
