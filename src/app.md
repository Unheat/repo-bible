# Codebase Bible

Paste a GitHub URL. The service ingests the repository, embeds the source, runs two AI agents over it, and produces a complete technical bible: a high-level architecture summary with a Mermaid diagram, plus a per-file deep-dive for every meaningful file in the codebase. The result is browsable in a dashboard and can be opened as a pull request on the source repository in one click.

The product is built for engineers who land in unfamiliar codebases and want to understand them in minutes instead of weeks.

~~~
There is no auth on this app. Single-tenant, hackathon-grade, and intended that way for the MVP. The repository list and every dashboard are visible to anyone with the URL. Adding auth is a tracked roadmap item but not part of MVP scope.

External services this app depends on:
- **GitHub REST API** — read-only for ingestion (file tree + raw content), Git Data API + Pull Requests API for the PR feature. Optional `GITHUB_TOKEN` secret raises the rate limit and unlocks private repos + PRs.
- **OpenAI Embeddings** via OpenRouter (`OPENAI_API_KEY` + `OPENAI_BASE_URL`). Model: `openai/text-embedding-3-small` (1536 dims). External because the MindStudio SDK has no embedding action.
- **Anthropic Claude 4.6 Sonnet** via the native `mindstudio.generateText` action. Used by both AI agents.

This hybrid is intentional: native SDK for everything the platform supports natively, external APIs only where the SDK has a gap.
~~~

## Two-Phase Pipeline

Every repository in the system passes through two independent phases, each with its own lifecycle status on the repository row.

**Phase 1: Ingestion.** The system fetches the file tree from GitHub, filters out boilerplate that does not benefit from documentation (lockfiles, build artifacts, binaries, minified bundles), persists the surviving files, downloads each file's raw source, splits it into overlapping chunks, and embeds every chunk into a 1536-dimensional vector. Embeddings are stored alongside the chunk text. This phase is purely structural — no LLM analysis happens here.

**Phase 2: Bible generation.** Two AI agents run in sequence. The Mapper produces a repository-level architecture summary plus a Mermaid flowchart of the primary data flow, grounded entirely in the file tree. The Deep-Dive then runs once per file with the architecture summary passed in as context, producing a senior-engineer-grade markdown writeup (file purpose, design patterns, public interface, internal logic walkthrough, dependencies, edge cases). All writeups land in a single documents table.

~~~
The two phases are deliberately independent. A repo can be ingested without ever generating a bible, and a bible can be re-generated without re-ingesting. This separation matters because:

1. Ingestion is comparatively cheap and stable — re-running it is rare unless the upstream repo changed materially.
2. Bible generation is expensive (one Mapper call + one Deep-Dive call per file). The user might want to re-run it after editing prompts, swapping models, or adding a new agent later. The split lets us re-run only the AI phase.

Each phase has its own status column on the repository row (`status` for ingestion, `docsStatus` for generation). Both run as fire-and-forget background tasks: the method that triggers them returns within a second after writing the row, then continues work in the background using JavaScript's persistent execution context. The frontend polls to track progress.
~~~

## Data Model

Four tables hold everything the system knows.

### Repositories

The top-level entity. One row per GitHub URL. Holds the canonical display name, both lifecycle statuses, and a timestamp of the last successful ingestion.

~~~
Stored in `dist/methods/src/tables/repositories.ts` as `Repositories`. Columns:

- `githubUrl` (string, unique) — full HTTPS URL like `https://github.com/owner/name`. Logical key. Unique constraint declared so `ingestRepository` can `upsert` instead of duplicating rows when a user pastes a URL we already know.
- `repoName` (string) — display name like `owner/name`. Derived from the URL once at ingest time and stored separately so the UI does not re-parse on every render.
- `status` (`'pending' | 'processing' | 'completed' | 'failed'`) — ingestion lifecycle. `pending` is reserved for future queueing and not used by current code. SQLite has no enums; the type alias enforces values at the call sites.
- `docsStatus` (`'idle' | 'generating' | 'completed' | 'failed'`, optional) — bible-generation lifecycle. Optional in the interface so existing rows that pre-date this column read as `undefined` and are treated as `idle` by call sites.
- `lastScannedAt` (number, optional, unix ms) — timestamp of the last successful ingestion. Deliberately not bumped on transitions to `processing` or `failed`. Answers "when did we last have a complete picture of this repo?" — not "when did we last touch the row?". Use the auto-managed `updated_at` for the latter.
~~~

### Files

One row per file that survived the boilerplate filter. References the repository.

~~~
Stored as `Files`. Columns:

- `repoId` (string) — FK into repositories.
- `filePath` (string) — relative path within the repo, e.g. `src/index.ts`.
- `language` (string) — detected from the file extension and a short list of special filenames (`Dockerfile`, `Makefile`, etc.). Stored as a lowercase token like `typescript`, `python`, `markdown`, `unknown`.

Unique constraint on `(repoId, filePath)` so re-ingestion of the same repo is idempotent on the files table.

Note: the schema does not currently prune stale files when a repo is re-ingested and a file was deleted upstream. Tracked as Out of Scope below.
~~~

### Code chunks

The chunk-and-embed output. Each row is one chunk of text plus its 1536-dim embedding. References a file.

~~~
Stored as `CodeChunks`. Columns:

- `fileId` (string) — FK into files.
- `chunkText` (string) — the raw text of the chunk.
- `chunkType` (`'code' | 'documentation' | 'mixed'`) — heuristic classifier from the chunker. Useful for retrieval ranking; not currently consumed by the AI pipeline.
- `embedding` (number array, stored as JSON) — 1536-dimensional float vector from `openai/text-embedding-3-small`.

This table exists to support a future retrieval-augmented Deep-Dive (cross-file chunk retrieval). The current Deep-Dive does not query embeddings — it works from the full file source plus the architecture context. Storing the chunks now means the retrieval upgrade is purely a method change later, not a schema change.
~~~

### Generated docs

All AI output. One row per generated document, discriminated by `docType`.

~~~
Stored as `GeneratedDocs`. Columns:

- `repoId` (string) — FK into repositories. Denormalized so a repo's full document set can be fetched in one query without joining through files.
- `fileId` (string, optional) — FK into files. Required for `file_deepdive` rows, absent for `repo_overview` rows.
- `docType` (`'repo_overview' | 'file_deepdive'`) — discriminator.
- `markdownContent` (string) — the full markdown writeup as produced by the model.

System columns provide `created_at` (used as the document's effective creation time — there is intentionally no separate `createdAt` column).

The discriminator pattern was chosen over two separate tables because both document types share the same lifecycle (created together by `generateBible`, read together by `getRepositoryDetail`) and the row count is small enough that scanning by repoId is fast.
~~~

## Methods

Five methods cover the full surface area.

### `ingestRepository`

Phase A (synchronous, returns within a few hundred ms): parse the URL, hit the GitHub REST API to fetch repo metadata, upsert the repository row with `status: 'processing'`, return `{ repositoryId, repoName, defaultBranch, status: 'processing' }`.

Phase B (background, fires after the synchronous return): fetch the full file tree, run the boilerplate filter, upsert one row per surviving file, then for each file fetch the raw content and run chunk + embed. On success the repository row flips to `status: 'completed'` and `lastScannedAt` is set. On any unhandled error the row flips to `status: 'failed'` and the error is logged to `console.error`.

~~~
Path: `dist/methods/src/ingestRepository.ts`. Exported as `ingestRepository`.

The fire-and-forget pattern: the method writes the `processing` row, then kicks off an un-awaited promise chain that does the heavy work, then returns. The execution environment persists across the return, so the background work continues. This is the canonical platform pattern documented in the methods reference.

The ingestion loop is **serial today**, not parallel. One file fetched at a time, one chunk + embed call at a time. This is fine for repos under a few hundred files but slow on large repos (the express smoke test took ~2.5 minutes for 208 files). Adding a concurrency pool here is tracked as Out of Scope.

The boilerplate filter is implemented as five layered rules in `dist/methods/src/common/fileFilters.ts`:
1. Drop files inside known boilerplate directories (`node_modules`, `dist`, `build`, `.git`, etc.).
2. Drop common lockfiles (`package-lock.json`, `yarn.lock`, `Cargo.lock`, etc.).
3. Drop files matching minification or sourcemap patterns (`*.min.js`, `*.map`).
4. Drop files with binary extensions (images, fonts, archives, executables).
5. Drop files larger than 1 MB.

Language detection (`detectLanguage`) maps file extension to a lowercase language token, with special cases for filenames that have no extension (`Dockerfile`, `Makefile`, `Procfile`, etc.).

The chunker (`dist/methods/src/common/textChunker.ts`) targets 4000-character chunks with 400-character overlap, splitting on logical boundaries (blank lines, then newlines, then sentence boundaries) before falling back to hard cuts. The classifier inspects each chunk and tags it `code`, `documentation`, or `mixed` based on the ratio of comment-like lines to code-like lines.

The embedder (`dist/methods/src/common/openaiClient.ts`) wraps the official `openai` SDK (v6 — v4 breaks our bundler because it pulls in node-fetch). Configured for OpenRouter via `OPENAI_BASE_URL`, but works against bare OpenAI by stripping the `openai/` prefix from the model ID. Resilience: a single 2-second-delayed retry on a 429 response, then null slots for the chunks that still failed (the surrounding code tolerates partial failures).
~~~

### `generateBible`

Phase A (synchronous): validate that the repository's `status === 'completed'`, flip `docsStatus` to `'generating'`, return `{ repositoryId, repoName, docsStatus: 'generating' }`.

Phase B (background): run the Mapper agent once over the full file tree, persist the resulting markdown as a `repo_overview` row. Then run the Deep-Dive agent once per file with the architecture summary passed in as context, persisting each result as a `file_deepdive` row. Deep-Dive calls run in a concurrency-8 pool. On success the row flips to `docsStatus: 'completed'`; on unhandled error it flips to `'failed'`.

~~~
Path: `dist/methods/src/generateBible.ts`. Exported as `generateBible`.

Both agents call `mindstudio.generateText` with `claude-4-6-sonnet`. Prompts and model config live in `dist/methods/src/common/llmPrompts.ts`:

- **Mapper** — temperature 0.1, `maxResponseTokens: 16000`, `config: { reasoning: 'true' }`. Adaptive thinking is enabled because architectural synthesis genuinely benefits from it; the cost is amortized over the whole repo (one call). The prompt instructs the model to ground every claim in a specific path from the file tree, use "likely" or "appears to be" for inferences, and refuse to reference paths not in the tree.

- **Deep-Dive** — temperature 0, `maxResponseTokens: 16000`, no reasoning override (default off). Code analysis is factual extraction; variance is a bug. The prompt enforces a CLOSE-WORLD CONSTRAINT: the model may only reference code that appears verbatim in the provided source, must use exact quotes in fenced code blocks rather than paraphrasing, and must say "implementation not in scope" rather than invent behavior of imported modules.

Both prompts explicitly forbid emojis, em dashes, and "Here is the analysis:" preludes. The Deep-Dive's output structure has seven required sections (File Purpose, Architecture and Design Patterns, Public Interface, Internal Logic Walkthrough, Dependencies and Integrations, Edge Cases and Error Handling, optional Observations).

Concurrency on the Deep-Dive loop: a simple promise pool of 8 workers. Each worker pulls the next file off the queue, generates the writeup, and pushes the result. Results are batch-inserted in groups of 50 to amortize database round-trips on large repos.

There is **no concurrent-run guard** on `generateBible`. If the user clicks Regenerate twice before the first run completes, two background pipelines run simultaneously, both writing to the same `generated_docs` table. The result is a mix of duplicated rows. Tracked as Out of Scope below; in practice the UI disables the button while `docsStatus === 'generating'`, so the only way to hit this is by direct API invocation.
~~~

### `listRepositories`

Returns every repository in the system as a flat array, sorted newest-first, each row enriched with summary stats: file count, document counts (overview + per-file), and both lifecycle statuses.

~~~
Path: `dist/methods/src/listRepositories.ts`. Returns `{ repositories: RepoSummary[] }`. The summary fields are `id`, `githubUrl`, `repoName`, `status`, `docsStatus` (defaulting `undefined` to `'idle'`), `lastScannedAt`, `fileCount`, `hasOverview`, `documentedFileCount`, `updatedAt`.

The home page polls this every 5 seconds. The implementation uses `db.batch` to fetch repositories, files (grouped by repo), and generated docs (grouped by repo) in a single round-trip rather than N+1 queries.
~~~

### `getRepositoryDetail`

Returns one repository's full state in a single round-trip: metadata, every file (alphabetical), the latest `repo_overview` document, and a map of fileId → latest `file_deepdive`.

~~~
Path: `dist/methods/src/getRepositoryDetail.ts`. Input: `{ repositoryId }`. Output: `{ id, githubUrl, repoName, status, docsStatus, lastScannedAt, files, overview, fileDocsByFileId }`.

`files` is an array of `{ id, filePath, language, hasDoc }` sorted alphabetically by `filePath`. `hasDoc` is true if a `file_deepdive` row exists for that file. The dashboard uses this flag to opacity-fade and disable click on file rows that don't yet have a writeup (during a partial-progress generating state).

`overview` is `null` if no `repo_overview` row exists yet, else the full markdown content + id + createdAt.

`fileDocsByFileId` is an object keyed by file id, value is the latest deep-dive doc for that file. Designed so the React component can do `O(1)` lookup when the user clicks a file in the sidebar — no hash map construction in render.

The dashboard polls this every 3 seconds while either status is non-terminal, then stops polling once both statuses settle.
~~~

### `openDocumentationPR`

Synchronous. Validates `docsStatus === 'completed'`, materializes the latest documents into a virtual file tree, runs the GitHub Git Data API tree-chain dance, opens a pull request on the source repository.

~~~
Path: `dist/methods/src/openDocumentationPR.ts`. Input: `{ repositoryId }`. Output: `{ prUrl, prNumber, branch, fileCount }`.

The PR contains a `docs/` directory with:
- `docs/ARCHITECTURE.md` — the latest `repo_overview` markdown.
- `docs/<original-path>.bible.md` — one file per `file_deepdive`, mirroring the original file's path under `docs/`.

GitHub Git Data flow:
1. Fetch the default branch's HEAD commit + its tree SHA.
2. Create one blob per documentation file (POST `/repos/:owner/:repo/git/blobs`). Concurrency 8 to keep this fast on large bibles.
3. Build a new tree pointing at those blobs, with `base_tree` set to the existing HEAD tree (so the PR is additive, not a wipe).
4. Create a new commit pointing at the new tree, with the HEAD commit as parent.
5. Create a new branch (`refs/heads/codebase-bible-{unix-ms}`) pointing at the new commit.
6. Open a pull request from the new branch back to the default branch.

PR title: `docs: Auto-Generated Codebase Bible`. Body has a brief description and a credit line.

Auth: requires `GITHUB_TOKEN` with `contents:write` and `pull_requests:write` on the target repo. A classic PAT with the `repo` scope works; a fine-grained PAT requires both Contents and Pull requests set to "Read and write."

Forks: not currently supported. The PR is opened against the source repo directly, which means the token holder needs write access on that repo. Fork-and-PR is tracked as Out of Scope.

The user-facing button on the dashboard ("Push to GitHub →") triggers this method. PR success surfaces as a toast in the bottom-right with the PR URL as a clickable link; failures surface as an error toast with the GitHub API message.

There is also a separate, longer-term roadmap idea: replace the in-app PR button with a "Download as HTML" flow that produces a single self-contained HTML bundle of the bible, letting users put it wherever they want without granting the app any GitHub write scopes. The PR feature stays for users who want one-click integration; the HTML download adds a no-auth alternative.
~~~

## Hybrid AI Routing

The pipeline uses two different model providers, accessed through two different clients.

**Embeddings → external (OpenRouter).** The MindStudio SDK does not currently expose an embeddings action. To get vector embeddings for code chunks, the system uses the official `openai` SDK pointed at OpenRouter via `OPENAI_BASE_URL` and `OPENAI_API_KEY`. Model: `openai/text-embedding-3-small` (1536 dimensions). The OpenRouter prefix is automatically stripped if the user configures bare OpenAI instead.

**Text generation → native (`mindstudio.generateText`).** Both AI agents call the platform's native text-generation action with `claude-4-6-sonnet`. No separate API key needed; credentials are handled by the execution environment.

~~~
This split is a deliberate architectural decision, not an accident.

The native SDK is preferred for everything it supports: zero credentials to manage, integrated rate limiting, integrated billing, and the platform handles routing and retries. The only reason to reach for an external client is when the SDK has a real gap.

Embeddings are that gap today. If the SDK adds an embeddings action later, the right move is to swap the OpenAI client for the native one. The chunk + embed code is isolated in `dist/methods/src/common/openaiClient.ts` precisely so this swap is a one-file change.

The `OPENAI_API_KEY` and `OPENAI_BASE_URL` secrets are configured in both dev and prod. The `GITHUB_TOKEN` secret is also configured in both, used by `openDocumentationPR` and (optionally) by the GitHub REST client during ingestion to raise the rate limit and unlock private repos.
~~~

## Out of Scope

Work explicitly deferred from the MVP. Each is a tracked roadmap item.

- **Stale file pruning.** Re-ingesting a repo never deletes file rows for files that no longer exist upstream. The orphaned rows linger in the table.
- **Concurrent ingestion within a repo.** The ingestion loop fetches and embeds one file at a time. A pool similar to the Deep-Dive's would cut large-repo ingestion time roughly 8x.
- **Concurrent-run guard on `generateBible`.** Two simultaneous runs against the same repo will both write to `generated_docs`, producing duplicates. The UI disables the button while generating, so the only way to hit this is direct API invocation, but it should still be guarded server-side.
- **Fork-and-PR for `openDocumentationPR`.** The PR is opened against the source repo directly, requiring write access. Forking and PR-from-fork would let any token holder document any public repo.
- **Retrieval-augmented Deep-Dive.** The Deep-Dive currently works from the full file source plus the architecture summary. A retrieval step that pulls in relevant chunks from other files would catch cross-file references.
- **Auth and multi-user.** No login, no roles, no per-user data. Adding auth opens the door to private bibles, sharing controls, and rate limiting per user.
- **Scenarios and seed data.** No scenarios are defined. Adding a "demo" scenario with a pre-ingested repo would let users see the dashboard without waiting for an ingest run.
- **Deep-link routing inside the dashboard.** The current URL is `/repos/:id`. The selected file or overview is local component state and not reflected in the URL, so it can't be shared or bookmarked.
- **Mobile-responsive dashboard.** The dashboard layout assumes desktop. Mobile is unstyled.
- **Download-as-HTML export.** A roadmap item to add a no-auth alternative to the GitHub PR flow: a single self-contained HTML bundle the user can host anywhere.
