Right now, sharing a bible means sharing a dashboard URL that requires the app to be running and the reader to land on the right state. A hosted bible is a public-facing read-only page: a clean URL that anyone can open, that renders the full bible without any app chrome, and that can be embedded in a team wiki, tweeted, or linked from a project README.

This is the moment Codebase Bible becomes a publishing tool, not just a generation tool.

## What it looks like

Every bible dashboard gains a "Share →" button that generates a hosted URL: `<app-domain>/b/<slug>`. The slug is derived from the repo name and a short hash. The hosted page looks identical to the dashboard reader but with no toolbar, no sidebar-based navigation replaced by a clean left nav, and an optional "Generated by Codebase Bible" footer with a "Generate one for your repo →" CTA.

The share URL is stable — it points to the bible as it was when published. Re-generating the bible produces a new version, and the user can choose to update the public URL to the latest version or leave the old one live.

Hosted bibles are indexed by search engines by default. An `og:title` and `og:description` are generated from the architecture overview. A robots tag is included.

## Key details

- **Publish action:** `publishBible({ repositoryId })` creates a `published_bibles` row with a slug, a snapshot of all generated docs at that moment, and a `publishedAt` timestamp.
- **Versioning:** each publish creates a new immutable snapshot. The `publishedBible` row tracks the `latest` pointer so updating the public URL is a pointer swap, not a content re-copy.
- **Privacy:** published bibles are public by default. A future auth layer can add access controls.
- **The hosted page:** a new route `/b/:slug` — no auth, no polling, renders the full bible from the published snapshot. Uses the same `MarkdownView` component. Left nav replaces the dashboard sidebar with a lightweight sticky nav.
- **The "Generate one for your repo" CTA** on every hosted page is the viral mechanic. A user lands on a hosted bible for a project they contribute to and immediately wants one for their own repo.
- **Custom slugs:** let users set a vanity slug (`/b/expressjs-express`) instead of the auto-generated one.

~~~
New table: `PublishedBibles`. Columns: `id`, `repositoryId` (FK), `slug` (unique), `snapshotOverview` (markdown string), `snapshotFileDocs` (JSON blob: `{ [filePath]: markdownContent }`), `publishedAt`, `repoName`, `githubUrl`.

The snapshot is a copy of the markdown at publish time, not a live join. This ensures the hosted URL never changes even if the bible is regenerated.

New route in the web interface: `/b/:slug`. Fetches the published bible via a new `getPublishedBible({ slug })` method (no auth required). Renders using the same MarkdownView component.

SEO: server-render the `<title>` and `<meta>` tags from the snapshot's repo name and architecture overview first paragraph. The rest of the page is client-rendered.
~~~
